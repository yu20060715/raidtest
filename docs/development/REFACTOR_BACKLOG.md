## BUGS

### B1 — stripe_volume_create omits raid_level initialization **FIXED**
**File:** `src/stripe_engine.c:66-189`  
**Detail:** `stripe_volume_create` does `memset(vol, 0, sizeof(STRIPE_VOLUME))` and never sets `vol->raid_level`.  
Zero init gives `RAID_LEVEL_NONE (0)`; `mirror_volume_create` correctly sets `RAID_LEVEL_MIRROR`.  
Stripe volumes report `RAID_LEVEL_NONE` on query. `stripe_volume_read/write` check `== RAID_LEVEL_MIRROR` to delegate, so stripe paths always run, but the metadata is wrong.  
**Fix:** Add `vol->raid_level = RAID_LEVEL_STRIPE;` after memset.  
**Resolution (2026-07-08):** Added `vol->raid_level = RAID_LEVEL_STRIPE;` at `src/stripe_engine.c:69` (after memset, before disk_count). Build PASS, 39/39 Tests PASS.

### B2 — mirror_volume_rebuild flush inside loop **FIXED**
**File:** `src/mirror_engine.c:164`  
**Detail:** `FlushFileBuffers(replacement->handle)` called every 64 MB chunk. For large volumes this issues thousands of flushes. One flush at the end is sufficient.  
**Fix:** Move flush outside the `while` loop.  
**Resolution (2026-07-08):** Moved `FlushFileBuffers` to after the while loop, guarded by `if (ok)`. `offset += chunk` moved inside loop before flush removal. Build PASS, 39/39 Tests PASS.

### B3 — mirror_volume_rebuild races with concurrent I/O on disks[] swap **FIXED**
**File:** `src/mirror_engine.c:174`  
**Detail:** `InterlockedExchangePointer(&vol->disks[replace_idx], replacement)` swaps a pointer that concurrent `stripe_volume_read/write` paths dereference without any lock. A read/write on the old disk handle could race with CloseHandle.  
**Fix:** Mark old disk unhealthy at rebuild start via existing `healthy` flag mechanism. Mirror I/O paths already check `healthy` before access, so they naturally skip the old disk during rebuild. On rebuild failure, rollback restores exact original healthy state.  
**Resolution (2026-07-08):** Added `old_was_healthy = InterlockedExchange(&old_disk->healthy, 0)` + conditional `healthy_count` decrement at rebuild start. On success, `InterlockedExchangePointer` + `replacement->healthy = 1` + increment (net count: -1 old +1 replacement = 0 delta if old was healthy, +1 if old was unhealthy). On failure (VirtualAlloc error, read/write/flush error): restore `old_disk->healthy = old_was_healthy`, conditional count increment. Added 3 tests: rebuild_with_concurrent_write (data integrity), rebuild_failure_rollback (state restoration), rebuild_healthy_count (count consistency).  
**Diff:** `src/mirror_engine.c` +17/-1 (healthy flag disabled + rollback), `src/test_mirror.c` +116 (3 new tests).  
**Verification:** Build PASS, 42/42 Tests PASS.

### B4 — stripe_volume_write error-path events[] / ovs[] consistency **FIXED**
**File:** `src/stripe_engine.c:574-602`  
**Detail:** Write path (non-cache) cleanup uses `CancelIoEx` and `CloseHandle` without NULL-guard on `events[j]`. When `CreateEventW` fails at index `i > 0`, all `events[0..i-1]` are valid and `ovs[0..i-1]` are memset. This is *de facto* safe because the loop sets `events[i]=evt` before `memset(&ovs[i])` on success, and the error loop only touches `j < i`. However the write-through cache path (line 518) *does* NULL-guard `events[j]`, creating inconsistency.  
**Fix:** Either add NULL-guards to both read and write error paths for defensive consistency, or remove them from the cache path.  
**Resolution (2026-07-08):** Added `if (events[j])` NULL-guard to all 4 non-cache error cleanup paths (read CreateEventW failure, read ReadFile failure, write CreateEventW failure, write WriteFile failure), matching the pre-existing write-through cache path pattern at line 520.  
**Diff:** `src/stripe_engine.c` — 4 locations, +4 lines (guard added before each CancelIoEx call).  
**Verification:** Build PASS, 39/39 Tests PASS.

### B5 — raid_cache_locked thread handle not tracked for safe cleanup **FIXED**

**Detail:** `_beginthreadex` result stored in `S()->vol.volume.cache.flush_thread` and `S()->cache.flush_thread`. If `cleanup_volume_cache` is called without first signalling the thread to stop, the thread may be terminated by `TerminateThread` (if that's what the cleanup does) or may outlive the volume. Thread destruction order is fragile.

**Diff:** `common.h` +1 field, `ram_cache.c` +2 (init+dbl-check), `ram_cache.c` 2 changes (loop condition), `cleanup.c` 1 change (signal flag).

### B11 — mirror_volume_write: cache-written but disk-not-written on write-through failure **FIXED**

**Resolution (2026-07-08):** Added `cache_invalidate()` API to `ram_cache.h`/`ram_cache.c` — clears `valid_map` + `dirty_map` bits for the affected offset range under `cache->lock`. **Recommendation:** Consider making `g_state` a parameter to all functions instead of a global. This would enable multiple instances, unit testing without side effects, and clearer data flow.

## CODE QUALITY NOTES

### CQ1 — Inconsistent error string patterns
Throughout: `LOG_OK`, `LOG_INFO`, `LOG_WARN`, `LOG_ERROR` use different argument styles. Some pass format strings, some pass pre-formatted strings.

### CQ2 — Magic numbers
- `64` hardcoded as max open files in fuse_bridge.c (line 35), repeated in statfs (line 504)
- `8` hardcoded as max selected disks in gui_panels.cpp (line 71, 1000)
- `256` hardcoded as max path length in many places
- `512 * 1024` hardcoded as event log max size in raid_service.c (line 38)
- `256 * 1024` hardcoded as event log trim target (line 42)

### CQ3 — Conditionally compiled code
- `fuse_bridge.c` is entirely `#ifdef USE_WINFSP` guarded
- `main.c` has `gui_run()` call without `#ifdef` — linking will fail if GUI isn't included

### CQ4 — 32/64-bit portability
- `DWORD` used for offsets in OVERLAPPED structures (line 457-458) — limits to 4GB file offsets
- `InterlockedExchangeAdd64` cast to `volatile LONG64*` with `LONG64` (which may differ from `LONGLONG`)
- `unsigned long long` used for printf of `uint64_t` — correct but not portable to all C99 libs

### CQ5 — Header dependency chain
`test_common.h` includes `stripe_engine.h` and `mirror_engine.h` directly. Any change to the volume structure forces recompilation of all tests. Consider forward declarations and a test-only header.
