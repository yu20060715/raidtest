# RAIDV3 Implementation Guide

目標只有一個：**把 RAIDV3 做成可以穩定展示的大學畢業專題正式專案**。
排序原則是：先保資料正確，再保流程一致，再保 GUI / CLI 真實可達，最後才做架構與文件整理。

---

## Phase 1

### Task 1 — Cache flush consistency and shutdown ordering

**Priority:** P0
**Risk:** High

1. **名稱**
   修正 RAM cache flush 一致性與關閉順序。

2. **為什麼要改**
   `cache_flush_all()` 目前有資料一致性風險：flush 過程中 dirty bit 的處理與實際寫入順序不夠安全，對畢業專題來說這是最不能留下的問題。
   這不是單純效能問題，是可能造成資料錯寫、舊資料覆蓋新資料的核心風險。

3. **哪些 source file**

   * `src/ram_cache.c`
   * `src/ram_cache.h`
   * `src/cleanup.c`
   * `src/raid_service.c`
   * `src/volume_manager.c`

4. **哪些 function**

   * `cache_init`
   * `cache_destroy`
   * `cache_write`
   * `cache_read`
   * `cache_flush_all`
   * `cache_flush_thread`
   * `cleanup_volume_cache`
   * `cleanup_cache`
   * `raid_cache_locked`

5. **哪些 struct**

   * `RAM_CACHE`
   * `STRIPE_VOLUME`

6. **哪些 global**

   * `g_state.cache.flush_thread`
   * `g_state.cache.cache_on`
   * `g_state.vol.volume.cache_enabled`
   * `g_state.vol.volume.cache_flush_in_progress`

7. **OpenCode 應該怎麼修改**
   不寫 code，只改策略：

   * 把 cache flush 的「dirty bit 清除時機」改成只在資料真正安全落盤後才視為完成。
   * flush thread 與 `cache off` / cleanup / destroy 的順序要統一，不能留下半關閉狀態。
   * 若 flush 失敗，必須保留 dirty 狀態，讓後續能重試。
   * 保留 cache 功能，但把一致性當成第一優先。

8. **修改完成後如何驗證**

   * **Build:** `build.bat`, `build_asan.bat`
   * **Tests:** `src/test_cache.c`, `src/test_journal.c`, `build_stress.bat` 相關壓力測試
   * **GUI:** 開啟 cache → 寫入資料 → 關閉 cache → 重新開啟 → 再寫入，確認流程穩定
   * **CLI:** `cache`, `cache off`, `cache wt`, `random`, `benchfs`

9. **修改風險**
   High

10. **優先等級**
    P0

### Task 2 — Canonicalize setup / restore / mount workflow

**Priority:** P1
**Risk:** High

1. **名稱**
   統一 quick setup、restore、mount 的流程語義。

2. **為什麼要改**
   現在 GUI 與 CLI 的 quick / restore 行為不完全一致。
   `quick` 在 CLI 會停下來等輸入，GUI quick setup 則是自動化工作流；`--auto`、`restore from config`、`restore from superblock` 之間也有語義差。
   這會讓教授在 demo 時一問就露餡。

3. **哪些 source file**

   * `src/main.c`
   * `src/cmd_handler.c`
   * `src/raid_service.c`
   * `src/volume_manager.c`
   * `src/config.c`
   * `src/wizard.c`
   * `src/gui.cpp`

4. **哪些 function**

   * `do_restore`
   * `cli_main`
   * `cmd_quick`
   * `cmd_process_auto`
   * `raid_quick`
   * `raid_load`
   * `raid_create`
   * `raid_mount`
   * `raid_config_save`
   * `raid_config_load`
   * `ShowWelcomeWizard`
   * `ShowRestoreWizard`
   * `W_QUICK_SETUP`
   * `W_LOAD_SUPERBLOCK`
   * `W_LOAD_CONFIG`

5. **哪些 struct**

   * `APP_CONFIG`
   * `APP_STATE`
   * `DISK_CONFIG`
   * `STRIPE_VOLUME`

6. **哪些 global**

   * `g_state`
   * `g_gui`

7. **OpenCode 應該怎麼修改**
   不寫 code，只改策略：

   * 定義一個「唯一的」quick setup 語義。
   * 若 quick 是自動化，GUI / CLI / auto 都應走同一條後端序列；若 quick 是互動式，就不要在文件與按鈕上宣稱是全自動。
   * restore 流程要先驗證磁碟數、再 create / mount / cache，避免半套復原。
   * 不要留下「看起來像完整復原，其實只做一部分」的路徑。

8. **修改完成後如何驗證**

   * **Build:** `build.bat`
   * **Tests:** 既有 test suite、restore / config 相關回歸測試
   * **GUI:** Welcome → Quick Setup、Restore Wizard、Rebuild Wizard 的完整走一次
   * **CLI:** `quick`, `--auto G`, `config-load`, `load`, `mount`, `unmount`

9. **修改風險**
   High

10. **優先等級**
    P1

## Keep / delete / defer summary

**現在就該保留並強化的：**

* RAID0 / RAID1 核心流程
* scan / select / init / create / mount / unmount / load
* benchmark / event log / diagnostics export
* planner 作為容量估算工具

**建議刪掉或隱藏的：**

* 沒有真正 UI 的 console state
* unreachable 的 Health Dashboard 與 Purge dialog，除非真的接線
* 未使用的 `metadata_upgrade`，如果近期不做版本遷移
* RC2 / RC4 混用
* 把 RAID10 寫成 runtime 支援的說法

**可以留到 v1.1 的：**

* 真正的 GUI console
* 額外 dashboard
* 不是 demo 必需的視覺美化
* 不影響功能正確性的文件潤飾
