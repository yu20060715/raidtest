# RAIDV3 Architecture Refactor Plan

> Generated by static analysis of the `raidv3` source tree.
> This document identifies architectural issues and ranks improvements by impact.

---

## Priority Legend

| Rank | Label | Meaning |
|------|-------|---------|
| P0 | **Critical** | Blocks stability, security, or onboarding |
| P1 | **High** | Significant maintainability or correctness risk |
| P2 | **Medium** | Improves modularity, reduces technical debt |
| P3 | **Low** | Nice-to-have cleaning |

---

## P0 â€” Critical

### 0.1 Separate the monolithic global state (`APP_STATE`)

**File:** `src/cmd_handler.h:34-42`

The entire application state lives in a single `extern APP_STATE g_state` global struct. Every module reads/writes into this monolith. This creates implicit coupling between all modules and makes unit testing impossible without full system state.

**Recommendation:** Decompose `APP_STATE` into owned contexts:
- `DiskPoolContext` (disk array, selection, pool sizes)
- `VolumeContext` (volume metadata, phases, cache reference)
- `RuntimeContext` (state machine, mount status)
- Pass contexts via function parameters, not globals.

**Impact:** Enables unit testing, clarifies data ownership, enables future multi-volume support.

---

### 0.2 State machine enforcement is shared between service layer and CLI layer

**File:** `src/raid_service.c:49` (`require()`), `src/cmd_handler.c:196-245` (`cmd_process`)

The `raid_service.c` functions enforce state machine transitions (e.g., `require(STATE_INITIALIZED)`).
But `cmd_process` in `cmd_handler.c` acquires `g_state_cs` and calls `raid_*` functions, while `main.c:11-61` (`do_restore`) also acquires `g_state_cs` and calls `raid_*` directly.
The critical-section discipline is inconsistent â€” some paths lock, others don't, and the state lock (`g_state_cs`) protects nothing specific.

**Recommendation:** Move state machine enforcement into a dedicated `StateMachine` module with explicit transition tables. Remove ad-hoc `require()` calls.

**Impact:** Eliminates state inconsistency bugs, clarifies thread-safety boundaries.

---

### 0.3 Cache/journal data-consistency race in `cache_flush_all`

**File:** `src/ram_cache.c:86-206`

The flush function:
1. Reads dirty blocks from cache into `flush_buffer` (under lock)
2. Writes `journal_data` entry (with the old data)
3. Clears dirty bits (under lock)
4. Writes `flush_buffer` to disk (outside lock)

Between steps 3 and 4, a concurrent write to the same cache block marks it dirty again with newer data. The physical I/O in step 4 writes the **old** data to disk. On crash recovery, the journal replays the old data, **overwriting the newer data** that was already on disk.

**Recommendation:** Restructure flush to either:
- Hold the cache lock through the physical I/O (reduces concurrency but is safe), or
- Use a two-phase commit: write new data to a pending-write buffer, atomically swap, then flush.

**Impact:** Eliminates silent data corruption on crash during cache flush.

---

### 0.4 No abstraction boundary between RAID engine and FUSE filesystem

**File:** `src/fuse_bridge.c` (entire file: 600 lines)

The FUSE layer directly accesses:
- `g_ctx.vol` pointer (a `STRIPE_VOLUME*`)
- `vol->cache.*`, `vol->bytes_written`, `vol->virtual_total_bytes`
- `vol->cache.lock`
- `g_open_files` file table

There is no volume/filesystem API. The FUSE callbacks implement caching, journal integration, and backpressure inline.

**Recommendation:** Define a `VolumeFs` interface:
```c
struct VolumeFs {
    int (*read)(void* ctx, char* buf, size_t size, off_t offset);
    int (*write)(void* ctx, const char* buf, size_t size, off_t offset);
    int (*flush)(void* ctx);
    int (*statfs)(void* ctx, struct fuse_statvfs* st);
};
```
Implement it once. `fuse_bridge.c` only mounts and dispatches.

**Impact:** Enables alternative FUSE backends, simplifies testing, decouples filesystem semantics from RAID internals.

## P1 â€” High

### 1.1 Split `stripe_engine.c` (712 lines) into focused modules

**File:** `src/stripe_engine.c`

This file contains:
- Volume creation (phase computation, capacity calculation)
- Volume I/O (read, write, map LBA)
- Volume expansion
- Testing helpers (`verify_io`, `random_test`)
- Mapping visualization (`dump_mapping`)

These are four distinct responsibilities.

**Recommendation:**
- `stripe_layout.c` â€” phase computation, capacity, normalize_ratios
- `stripe_io.c` â€” read, write, map_lba, map_single_byte
- `stripe_expand.c` â€” volume expansion
- `stripe_debug.c` â€” dump_mapping, verify_io, random_test (test utils should move to test files)

**Impact:** Each module becomes testable independently. Volume creation logic is not coupled to I/O paths.

---

### 1.2 Split `raid_service.c` (610 lines) â€” it is the "god object" facade

**File:** `src/raid_service.c`

All 30+ public operations are exposed from a single file. This includes:
- Lifecycle (init/cleanup)
- Discovery (scan/select/bench)
- Pool management (5 separate init_pools helpers)
- Volume operations (create/mirror/mount/unmount/destroy/expand/rebuild)
- Cache operations (on/off/wt)
- Config persistence
- Tests and benchmarks
- Quick setup wizard integration

**Recommendation:** Group into domain facades:
- `raid_discovery.c` â€” scan, select, mapdrive, bench
- `raid_pool.c` â€” init_pools (all helpers)
- `raid_volume.c` â€” create, mirror, expand, rebuild, mount, unmount, load, destroy, purge
- `raid_cache.c` â€” cache operations
- `raid_config.c` â€” config save/load
- `raid_actions.c` â€” test, random, benchfs, simulate, wizard, quick

**Impact:** Each domain is cohesive, has clear dependencies, and can be tested in isolation.

---

### 1.3 Split `fuse_bridge.c` (600 lines) â€” extract file table and I/O helpers

**File:** `src/fuse_bridge.c`

The file contains:
- In-memory file table (open file tracking)
- Directory hierarchy helpers (parent_dir_exists, is_direct_child, is_dir_empty)
- Cache backpressure logic
- All 15 FUSE operations
- Mount/unmount thread management

**Recommendation:** Extract into:
- `fuse_filetable.c/h` â€” file table CRUD, directory traversal
- `fuse_cache_helpers.c/h` â€” backpressure, highest-written tracking
- Keep pure FUSE operations and mount dispatch in `fuse_bridge.c`

**Impact:** File table can be unit-tested. Cache helpers become reusable.

---

### 1.4 Extract superblock version parsing from monolithic reader

**File:** `src/superblock.c:167-307`

`try_read_superblock_from_drive` handles v1, v2, v3, and v4 in one giant function with deeply nested `if/else` blocks. Adding a v5 would require touching this one function.

**Recommendation:** Use a strategy pattern or a dispatch table:
```c
typedef bool (*sb_reader_fn)(const uint8_t* raw, size_t raw_len, SUPERBLOCK* out);
static sb_reader_fn g_readers[] = { NULL, read_v1, read_v2, read_v3, read_v4 };
```

**Impact:** Each version's reader is independently testable. Adding new formats doesn't risk breaking old ones.

---

### 1.5 Remove `#include` of `.c` files in `raid_service.c`

**File:** `src/raid_service.c` (includes test source files)

```c
#include "tests/test_stripe.c"
#include "tests/test_mirror.c"
#include "tests/test_cache.c"
#include "tests/test_journal.c"
#include "tests/test_concurrent.c"
#include "tests/test_random_io.c"
```

This is a severe layering violation. Test code is compiled into the production binary.

**Recommendation:** Define a test registration API (e.g., `test_register_all()` declared in a header, implemented in a test helper). Remove `#include` of `.c` files entirely.

**Impact:** Production binary no longer contains test code. Link-time dependencies are clarified.

---

### 1.6 `gui.h` lacks include guard

**File:** `src/gui.h`

```c
// no #pragma once, no #ifndef guard
RC gui_run(void);
```

Double inclusion causes redefinition error.

**Recommendation:** Add `#pragma once`.

**Impact:** Prevents potential compilation failure on certain include orders.

---

### 1.7 Threading model is ad-hoc

Multiple synchronization mechanisms are used inconsistently:
- `g_state_cs` â€” global critical section (cmd_handler.h)
- `vol->cache.lock` â€” per-cache critical section (ram_cache.c)
- `g_file_table_lock` â€” per-FUSE-bridge critical section (fuse_bridge.c)
- `g_eb_cs` â€” event bus critical section (event_bus.c)
- `InterlockedExchange` spinlock (`g_flush_in_progress`, fuse_bridge.c:24)
- `volatile LONG` + `Interlocked*` for counters (profiler.c, fuse_bridge.c)

Lock ordering is not documented. The `event_bus_publish` function drops `g_eb_cs` before calling callbacks, which could cause subtle reentrancy issues.

**Recommendation:** Document a single lock hierarchy. Consider consolidating into a single lock (or using SRWLOCK on Windows). Remove the `g_flush_in_progress` spinlock in favor of a real mutex.

**Impact:** Eliminates deadlock risk. Makes thread safety auditable.

## P2 â€” Medium

### 2.1 `bench_io.c` (211 lines) uses complex macro for benchmarking

**File:** `src/bench_io.c:62-108`

The `DO_BATCH` macro spans 47 lines and contains control flow (`for`, `if`, variable declarations). This makes debugging and profiling impossible.

**Recommendation:** Replace with a static helper function taking parameters for pass index.

**Impact:** Readable, debuggable benchmark code.

---

### 2.2 `cleanup.c` (152 lines) mixes concerns

**File:** `src/cleanup.c`

`cleanup_all()` calls `cleanup_scan_all_drives()` which does a brute-force scan of A-Z drives attempting to delete pool files, benchmarks, and directories. This is destructive cleanup that should be a separate maintenance tool, not part of the normal cleanup path.

**Recommendation:** Separate into:
- `cleanup_session.c` â€” normal lifecycle cleanup (cache, volume, session)
- `cleanup_aggressive.c` â€” brute-force drive scanning cleanup (for `--cleanup` flag only)

**Impact:** Normal exit won't accidentally delete pool files.

---

### 2.3 `daemon.c` (324 lines) has duplicated wait loops

**File:** `src/daemon.c`

`daemon_run` (line 146) and `daemon_main` (line 176) have ~90% duplicated logic: both call `daemon_load_volume`, mount, wait for stop, then cleanup. The only difference is how they wait (stdin pipe vs. pure event).

**Recommendation:** Unify into a single `daemon_loop` that accepts a wait-strategy callback.

**Impact:** Reduces duplication. Single code path for daemon and service modes.

---

### 2.4 `disk_scanner.c` (185 lines) uses `realloc` in a grow loop

**File:** `src/disk_scanner.c:43`

The pattern:
```c
void* new_disks = realloc(disks, (count + 1) * sizeof(DISK_INFO));
if (!new_disks) { CloseHandle(h); continue; }
disks = (DISK_INFO*)new_disks;
```
While functionally correct (original pointer preserved on failure), this leaks intermediate allocations on realloc failure.

**Recommendation:** Pre-allocate with `calloc` for max expected count (e.g., 26 drives), or use a linked-list during enumeration and convert to array at the end.

**Impact:** Eliminates allocation-failure edge case. Simplifies error handling.

---

### 2.5 Mirror engine I/O has no timeout/fail-fast

**File:** `src/mirror_engine.c:33-74`

When reading from a mirror, the function tries each healthy disk sequentially. If a disk is slow (not faulty, just slow), the read blocks until the OS timeout. There is no configurable timeout or fail-fast mechanism.

**Recommendation:** Use overlapped I/O with a timeout in the mirror read path, or track latency and deprioritize slow disks.

**Impact:** Mirror read performance degrades gracefully instead of hanging.

---

### 2.6 `journal.c` allocates unbounded memory for recovery

**File:** `src/journal.c:85-185`

`journal_recover_all` reads the entire journal file into memory with `malloc(file_size.QuadPart)`. A maliciously large or corrupted journal file could cause memory exhaustion.

**Recommendation:** Stream-parse the journal in fixed-size chunks, or cap the maximum journal size at creation.

**Impact:** Prevents OOM from corrupted/malicious journal files.

---

### 2.7 `config.c` uses ad-hoc JSON parsing

**File:** `src/config.c:60-113`

The config "parser" uses `wcsstr` + `swscanf` line-by-line. It doesn't handle:
- Nested objects
- Escaped characters
- Whitespace variations
- Malformed JSON

**Recommendation:** Use a minimal JSON library (e.g., `cJSON` or `json-parser`) or the Windows `IXmlReader` for XML.

**Impact:** Robust config parsing. No silent misinterpretation of config values.

---

### 2.8 `cmd_handler.c` (260 lines) command dispatch is a chain of `strcmp`

**File:** `src/cmd_handler.c:207-240`

```c
if (strcmp(args[0], "scan") == 0) rc = cmd_scan();
else if (strcmp(args[0], "mapdrive") == 0) rc = cmd_mapdrive(argc - 1, args + 1);
// ... 27 more else-if branches
```

**Recommendation:** Use a dispatch table:
```c
typedef RC (*cmd_fn)(int argc, char* argv[]);
typedef struct { const char* name; cmd_fn fn; } Command;
static const Command commands[] = {
    { "scan",     cmd_scan },
    { "mapdrive", cmd_mapdrive },
    // ...
};
```

**Impact:** O(1) dispatch. Adding a new command requires one table entry instead of three edits.

## P3 â€” Low

### 3.1 Rename `STRIPE_VOLUME` to `RAID_VOLUME`

**File:** `src/common.h:217`

The struct represents both stripe (RAID0) and mirror (RAID1) volumes. The name `STRIPE_VOLUME` is misleading.

**Impact:** Code clarity.

---

### 3.2 `metadata_manager.c` is a pass-through facade

**File:** `src/metadata_manager.c:1-28`

Every function delegates directly to `superblock_*` or `event_bus_*`. The file provides no abstraction, state, or transformation.

**Recommendation:** Either give `metadata_manager` real responsibility (caching, validation, conflict resolution) or remove it and call superblock functions directly.

**Impact:** Removes indirection. Clarifies API ownership.

---

### 3.3 `planner_engine.c` contains printing logic

**File:** `src/planner_engine.c:47-92`

`planner_print` mixes calculation results with formatted output. This prevents reuse in the GUI or programmatic consumers.

**Recommendation:** Have `planner_calculate` return a result struct. Let callers format as needed.

**Impact:** The GUI can use the planner without parsing printf output. Testable.

---

### 3.4 Inconsistent naming conventions

- Some files use `snake_case` functions (`stripe_volume_create`), others use mixed or abbreviated names (`cmd_init_all`, `raid_init_pools`).
- Some files use `g_` prefix for globals, others don't.
- Header file names inconsistently match source file names (already matched, but some are only headers with no .c).

**Recommendation:** Adopt a single naming convention (e.g., `module_action_object`).

---

### 3.5 `test_runner.c` is a stub

**File:** `src/test_runner.c:1-4`

```c
int main(void) { return 0; }
```

**Recommendation:** Either implement a proper test runner or remove the file.

---

### 3.6 `raid_simulate()` is a stub with side effects

**File:** `src/raid_service.c:511-549`

The function has full behavior (disk fail/healthy/disconnect simulation) but is declared in the header alongside real operations.

**Recommendation:** Move to a `debug_helpers.c` module or gate with `#ifdef DEBUG`.
