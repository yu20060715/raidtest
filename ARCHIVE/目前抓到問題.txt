# BUG_AUDIT.md

## 結論

最重要的是 **P0 cache flush race**。
其他問題多半是「功能不完整 / 路徑不一致 / thread safety 不夠乾淨」，但這一個會直接碰到資料正確性。

## P0

### `src/ram_cache.c::cache_flush_all()`

* 問題：

  * 這裡在持有 cache lock 時，把 dirty block 複製到 flush buffer，然後就**先清 dirty bit**，再把資料真正寫到磁碟。
  * 這表示另一個 thread 可能在 flush 完成前又寫入同一批 block。
  * 結果可能是：**舊資料覆蓋新資料**，造成資料不一致，嚴重時就是資料損失。
* 影響：

  * 高風險、實際可見、不是單純 log 問題。

## P1

### `auto_restore_or_quick()` 的 restore 只會處理第一顆 disk

* 問題：

  * `src/cmd_handler.c` 的 `auto_restore_or_quick()` 迴圈裡立刻 `break`，只 map 第一顆 disk。
  * 這讓 auto-restore 路徑變成「半 restore」。
* 影響：

  * 使用者以為是完整 restore，實際上可能只有第一顆被接上。

### GUI worker thread 與 UI thread 共享狀態沒有完整鎖

* 問題：

  * `g_gui` 裡很多欄位在 worker thread 直接寫、render thread 直接讀。
  * `worker_running / worker_done / progress_frac` 有 atomic / volatile，但大量字串與結構欄位沒有完整保護。
* 影響：

  * UI race、顯示錯亂、偶發未定義行為。

## P2

### `raid_quick()` / GUI Quick Setup 對部分失敗不夠嚴格

* 問題：

  * 某些 cache 或 mount 失敗沒有完整回傳失敗狀態。
  * GUI worker 會部分報告成功，但其實流程沒有完全完成。
* 影響：

  * 容易在 demo 時誤判成功。

### `raid_simulate('d')` 之後的 `healthy` 恢復不是真正恢復

* 問題：

  * disconnect 模式會 close pool file、把 disk 標成 faulty。
  * `healthy` 模式只改旗標，沒有把 handle 重新打開。
* 影響：

  * `simulate healthy` 比較像假恢復。

### `raid_destroy()` 會重複發 `EVENT_VOLUME_DESTROYED`

* 問題：

  * `raid_destroy()` 自己 publish 一次，`volume_destroy()` 又 publish 一次。
* 影響：

  * 會造成重複 log / 重複 event side effect。

### `ShowWelcomeWizard()` 的 “Quick Setup (Recommended)” 命名誤導

* 問題：

  * 按鈕只是關閉歡迎視窗，不是真的執行 quick setup。
* 影響：

  * demo / 教授實測時容易被問到。

## P3

### 版本字串不一致

* `src/cmd_handler.c` banner 還印 `RC2`
* `src/main.c` / `src/gui.cpp` 是 `RC4`
* 影響：

  * 文件與執行檔資訊混亂，但不會直接壞掉。

### 文件與 source 的版本 / 狀態標示落差

* 例如 README 還有 RC2 badge，但 code 已經是 RC4。
* 影響：

  * 會讓評審覺得專案整理不夠乾淨。

## 資源 / memory leak / handle leak

* 我沒有看到很明顯、可直接證實的「持續性 handle leak」比 P0 cache race 更嚴重的東西。
* 大多數 thread / handle / device 釋放路徑都有做收尾。
* 目前真正的核心風險是**資料一致性與 thread safety**，不是典型 memory leak。

# UNIVERSITY_REVIEW.md

## 結論

**這個專案已經夠做大學畢業專題。**
理由很直接：它不是 demo 程式，而是有完整系統輪廓的 Windows RAID 工具，包含：

* 磁碟探索
* RAID0 / RAID1
* superblock / journal
* RAM cache
* WinFsp 掛載
* GUI + CLI + daemon + service
* config 持久化
* event log
* benchmark / diagnostics / wizard

這些加在一起，專題厚度是夠的。

## 教授最可能追問的點

* 你的 cache 一致性怎麼保證？
* power loss / partial write / rollback 怎麼處理？
* WinFsp 為什麼能代表 RAID？它的限制是什麼？
* RAID1 rebuild 的流程怎麼驗證？
* GUI worker thread 與 background flush thread 怎麼同步？
* quick / restore / load / config-save 的差別是什麼？
* benchmark 測的是 RAID engine 還是 mount layer？
* 失敗情境下有沒有自動恢復或最小損失保證？

## 會扣分的地方

* `Health Dashboard`、`Confirm Purge` 這類 dead UI
* 版本字串與文件狀態不一致
* `quick` / restore 這些流程有部分語義不夠一致
* 測試證據還不夠完整
* 有一個真的會傷資料的一致性風險（P0）

## 亮點

* 不是單純把功能堆在一起，而是真的有 state machine、event bus、journal、superblock
* GUI 與 CLI 都不是空殼
* 有 service / daemon / mount layer，專題完整度高
* 有 developer / beginner 分層的 UI 思路

## 還差最後一步

* 把 dead UI 刪乾淨或接回去
* 修 cache flush race
* 修 restore / quick flow 的不一致
* 把文件版本號、功能描述、實際行為對齊
* 補可重現的測試與 demo script
