## P0 — Data Loss / Crash / Security

### P0-01: `cache_flush_all` — stale snapshot replayed after concurrent write

**File:** `src/ram_cache.c:124-133`

The cache lock is released between `memcpy` to `flush_buffer` (line 124) and `journal_data` (line 128). A concurrent `cache_write` can:
1. Acquire the lock, write new data to the same blocks, set dirty bits (lines 56-63)
2. Release the lock
3. `journal_data` writes the **stale** `flush_buffer` snapshot

Then dirty bits for this batch are cleared unconditionally (lines 132-133), including bits the concurrent writer set. If the process crashes between the dirty-bit clear and the physical I/O, the latest write is lost. Journal replay restores the stale snapshot.

**Fix:** Hold the cache lock across `memcpy` → `journal_data` → dirty-bit clear, or snapshot dirty-bit ownership.

### P0-02: Stack buffer overflow in `parent_dir_exists`

**File:** `src/fuse_bridge.c:103-107`

```c
char parent[256];
size_t plen = (size_t)(slash - path);
strncpy(parent, path, plen);   // overflow when plen > 255
parent[plen] = 0;              // write past end
```

`plen` is unbounded. A path longer than 255 chars before `/` overflows the stack buffer. Reachable via `raid_open` (line 176), which lacks the `strlen(p) > 255` guard that `raid_create` and `raid_mkdir` have.

**Fix:** Guard `plen >= sizeof(parent)` before `strncpy`.

---

### P0-03: Stack buffer overflow in `raid_rename` — `wsrc`

**File:** `src/fuse_bridge.c:269`

```c
wchar_t wsrc[256];
mbstowcs(wsrc, src, 256);      // fills wsrc[0..254] + L'\0' at [255]
wcscat(wsrc, L"/");            // writes L'/' at [255], L'\0' at [256] → overflow
```

When `wcslen(wsrc) == 255`, the `wcscat` write past the buffer by 2 bytes.

**Fix:** Check `wcslen(wsrc) < 255` before appending.

---

### P0-04: Stack buffer overflow in `raid_rename` — `newpath`

**File:** `src/fuse_bridge.c:273-278`

```c
wchar_t newpath[256];
wcscpy(newpath, wdst);         // up to 255 chars
wcscat(newpath, L"/");         // up to 256 chars
wcscat(newpath, suffix);       // suffix up to ~255 chars → total ~511 into buffer of 256
```

Unbounded concatenation overflows the stack buffer by a large margin.

**Fix:** Use `wcsncpy`/`wcsncat` with size limit, or reject paths that would overflow.

### P0-05: No synchronization in `raid_service.c` — 34 functions race on `g_state`

**File:** `src/raid_service.c` (all 34 functions)

Zero calls to `gs_lock()` / `gs_unlock()` in any function. The global critical section `g_state_cs` is initialized here (line 59) but never acquired. Every `S()` access is a data race when called from FUSE callbacks, GUI thread, CLI handler, or daemon threads concurrently.

Other files (`daemon.c`, `cmd_handler.c`, `main.c`) DO use `gs_lock/gs_unlock` before calling these functions, but there is no guarantee all call paths are covered, and no documentation of which caller owns the lock.

**Fix:** Acquire `g_state_cs` at the start of every public `raid_*` function.

---

### P0-06: Stack buffer overflow in JSON parser

**File:** `src/config.c:86`

```c
char lang[8] = {0};
swscanf(p, L" \"language\": \"%[^\"]\",", lang);   // no width limit
```

The `%[^\"]` format specifier has no width. A `language` value longer than 7 characters overflows the 8-byte stack buffer. Exploitable via crafted `config.json`.

**Fix:** Use `%7[^\"]` to limit writes.

---

### P0-07: Unbounded `malloc` from untrusted journal file size

**File:** `src/journal.c:100`

```c
uint8_t* raw = (uint8_t*)malloc((size_t)file_size.QuadPart);
```

`file_size` comes directly from `GetFileSizeEx` on a user-writable file. No sanity check. A crafted journal with a large declared size causes OOM crash. On 32-bit, truncation in `(size_t)` and `(DWORD)` for `ReadFile` can mismatch.

**Fix:** Reject journal sizes beyond a reasonable maximum (e.g., 64 MB).

## P1 — Significant Correctness Risk

### P1-01: `cache_flush_all` is not re-entrant — concurrent calls corrupt state

**File:** `src/ram_cache.c:86` (callers at `ram_cache.c:229`, `bench_io.c:188`, `cleanup.c:13`, `fuse_bridge.c:385,451,463`, `test_cache.c:153`, `test_mirror.c:161`)

No guard against concurrent entry. Two threads in `cache_flush_all` will:
- Clobber each other's `flush_buffer` contents (line 124)
- Interleave journal entries (multiple `JT_BEGIN` without matching `JT_COMMIT`)
- Race on `cache_flush_in_progress` (lines 137, 192)

**Fix:** Use `InterlockedCompareExchange` on an atomic flag at function entry to reject concurrent calls.

---

### P1-02: `journal_begin`/`journal_data`/`journal_commit` return values discarded

**File:** `src/ram_cache.c:93,128,199`

All three journal calls have their `bool` return values silently ignored. If `journal_data` fails (disk full, I/O error), dirty bits are still cleared (lines 132-133) and physical I/O proceeds. On subsequent crash, the journal is incomplete and the batch is unrecoverable → **data loss**.

**Fix:** Check return values. On failure, do not clear dirty bits.

---

### P1-03: Dirty bits cleared without ownership tracking

**File:** `src/ram_cache.c:132-133`

```c
for (uint32_t i = b; i < b + run; i++)
    cache->dirty_map[i / 8] &= ~(1 << (i % 8));
```

Dirty bits are cleared unconditionally for the batch range, regardless of whether a concurrent `cache_write` set additional bits on the same blocks between the snapshot and the re-lock (see P0-01). This loses the concurrent writer's dirty marking.

**Fix:** Snapshot dirty bits at selection time; only clear bits that were in the snapshot.

---

### P1-04: `cache_flush_in_progress` set after dirty-bit clear

**File:** `src/ram_cache.c:131-137`

The `InterlockedExchange(&vol->cache_flush_in_progress, 1)` happens after dirty bits are cleared. Between the clear (line 132-133) and the flag set (line 137), a concurrent write sees `cache_flush_in_progress == 0`, writes to cache, and its dirty bits are then cleared by the flush.

**Fix:** Set `cache_flush_in_progress = 1` before clearing dirty bits, while the cache lock is still held.

---

### P1-05: No I/O timeout in mirror engine

**File:** `src/mirror_engine.c:63,88,151,155`

All calls to `stripe_read_raw`/`stripe_write_raw` have no timeout. A hung disk blocks the thread indefinitely — no alertable wait, no cancel-after-timeout.

**Fix:** Use overlapped I/O with `WaitForMultipleObjects` and a timeout.

---

### P1-06: Missing `src` length check in `raid_rename`

**File:** `src/fuse_bridge.c:254-255`

Only `strlen(dst) > 255` is checked. `src` is not validated. A long `src` is silently truncated by `mbstowcs`, causing incorrect prefix matching.

**Fix:** Add `if (strlen(src) > 255) return -ENAMETOOLONG`.

---

### P1-07: Missing path length check in `raid_open`

**File:** `src/fuse_bridge.c:172-191`

Unlike `raid_create` (line 197) and `raid_mkdir` (line 214), `raid_open` does not guard against paths >255 chars, enabling the `parent_dir_exists` overflow (P0-02).

**Fix:** Add `if (strlen(p) > 255) return -ENAMETOOLONG`.

---

## P2 — Minor Correctness / Performance

### P2-01: O(n^2) `realloc` in disk scanner

**File:** `src/disk_scanner.c:43`

```c
void* new_disks = realloc(disks, (count + 1) * sizeof(DISK_INFO));
```

Single-element growth in a loop causes O(n^2) copy cost. Minor for typical cases (few drives) but pathological with many devices.

**Fix:** Double the allocation size on each grow, or pre-scan count.

---

### P2-02: Non-atomic release of profiler slot

**File:** `src/profiler.c:45,78`

```c
g_profiler.samples[slot].active = false;   // should be InterlockedExchange
```

Slot claimed atomically via `InterlockedCompareExchange` (line 28) but released with a plain write. No memory barrier between the read of `start_time` and the release of the slot.

**Fix:** Use `InterlockedExchange(&g_profiler.samples[slot].active, 0)`.

---

### P2-03: `lpNumberOfBytesWritten` should be `NULL` for overlapped I/O

**File:** `src/ram_cache.c:163`

```c
DWORD dummy = 0;
WriteFile(..., &dummy, &ovs[e]);
```

Passing non-NULL `lpNumberOfBytesWritten` with overlapped I/O is ignored per MSDN, but can cause access violation on some Windows versions with manual-reset events.

**Fix:** Pass `NULL`.

---

### P2-04: `running` flag not set atomically in `cache_destroy`

**File:** `src/ram_cache.c:38`

```c
cache->running = 0;   // should be InterlockedExchange
```

Inconsistent with the rest of the codebase which uses `InterlockedExchange` for atomic flags (`healthy`, `faulty`, `cache_flush_in_progress`). Safe on x86 but not on ARM.

**Fix:** Use `InterlockedExchange(&cache->running, 0)`.

---

### P2-05: Duplicate code between `daemon_run` and `daemon_main`

**File:** `src/daemon.c:148,188` (volume load), `151-156,197-202` (mount), `165-168,216-218` (cleanup), `170-172,220-224` (shutdown)

`daemon_run` is a stripped-down copy of `daemon_main`. Four code blocks duplicated verbatim. Maintenance hazard — fixes must be applied in two places.

**Fix:** Extract shared logic into a helper function called by both paths.

---

### P2-06: Unnecessary `InterlockedIncrement` inside critical section

**File:** `src/fuse_bridge.c:67`

```c
InterlockedIncrement(&g_open_count);  // already inside EnterCriticalSection
```

File-table lock is already held. Plain `g_open_count++` is sufficient and consistent with decrements (lines 84, 246) which use `--g_open_count` under the same lock.

**Fix:** Use plain increment.

## P3 — Cosmetic / Informational

### P3-01: Dangling `JT_BEGIN` when all blocks cleaned between begin and lock

**File:** `src/ram_cache.c:93,199`

`journal_begin` (line 93) is called before acquiring the cache lock. If a concurrent `cache_write` dirties and then another thread flushes all blocks between lines 93 and 97, the loop finds nothing dirty, `flushed` stays 0, and `journal_commit` is skipped — leaving a dangling `JT_BEGIN` in the journal. Harmless (recovery ignores incomplete transactions), but wasteful.

**Fix:** Move `journal_begin` inside the lock, or skip it when no dirty blocks.

---

### P3-02: `raid_service.c` defines its own `S()` macro instead of using `cmd_handler.h`'s extern

**File:** `src/raid_service.c:13`

```c
static APP_STATE* S(void) { return &g_state; }
```

Duplicates the accessor pattern. `cmd_handler.h` already declares `extern APP_STATE g_state`. Slight maintenance risk if the accessor logic diverges.

**Fix:** Use the common `S()` from a shared header, or remove `static` to expose to tests.

---

### P3-03: No bounds check on `plen == 0` early return order in `parent_dir_exists`

**File:** `src/fuse_bridge.c:105`

```c
if (plen == 0) return true;   // after strncpy, not before
```

If `plen == 0`, `strncpy(parent, path, 0)` writes nothing and `parent[0] = 0` is correct. The early return is redundant but harmless. Should be moved before `strncpy` for clarity.
